* 실제 Unity 프로젝트 경로는 기본적으로 다음으로 본다.

```text
C:\Users\alstj\Documents\GitHub\Capstone-ObjHead
```

* 이 프로젝트는 2D 턴제 포격 게임이며, 지형 파괴/생성, 캐릭터 머리 스킬, 공용 머리 아이템, 턴 순환, 카메라 조작이 핵심이다.

---

## 작업 전 확인 규칙

* 작업 전에 사용자가 준 텍스트, 기획서, 요구사항 파일을 먼저 읽고 구현 범위를 파악한 뒤 진행한다.
* 사용자가 업로드한 zip, txt, md, cs, unitypackage, 이미지 등의 내용을 가능한 먼저 확인한다.
* 기존 구조를 추측으로 갈아엎지 말고, 현재 프로젝트에 있는 스크립트와 씬 구조를 확인한 뒤 수정한다.
* 사용자가 직접 만든 변경사항은 되돌리지 않는다.
* 큰 포맷팅 변경, 파일 전체 재정렬, 불필요한 리네임은 피한다.
* 한글 경로, 한글 파일명, 비UTF-8 파일을 다룰 때 인코딩 깨짐에 주의한다.

---

## 작업 우선순위

* 단순 설명보다 실제 프로젝트에 반영되는 구현을 우선한다.
* 로컬 시연 및 프로토타입 기능은 서버 연동보다 Unity 클라이언트에서 먼저 눈에 보이는 형태를 우선한다.
* 지형 기능은 실제 플레이에서 시연 가능해야 하므로, 코드만 존재하고 씬에서 동작하지 않는 상태로 끝내지 않는다.
* 새 기능은 가능하면 Inspector에서 조절 가능한 `[SerializeField]` 값으로 분리한다.
* Unity UI는 가능하면 TextMeshPro와 `[SerializeField]` 기반 Inspector 연결을 사용한다.

## 2파트 지형/맵 담당 수정 가능 범위

2파트 지형/맵 담당 범위에서는 다음 파일과 인접 시스템을 수정할 수 있다.

* `TerrainManager`
* `TerrainChunk`
* Terrain collision mask
* 지형 생성/파괴 로직
* Dirty Chunk Collider 갱신
* 캐릭터 스폰
* 공용 머리 스폰
* WaterZone / DeathZone
* 카메라 Bounds 및 카메라 이동
* 조준/차징처럼 지형 기능 시연에 필요한 인접 코드
* 턴 흐름 중 지형 생성/파괴 시연에 필요한 최소 인접 코드

단, 전투 전체 구조를 갈아엎거나 서버/멀티플레이어 코드를 건드리는 작업은 사용자가 명시적으로 요청했을 때만 한다.

---

## 서버 및 멀티플레이어 코드 보호

* `UDPServer.cs`, `Network`, `Server`, `Multiplayer`, UDP 송수신, 패킷, 방 관리 관련 코드는 사용자가 명시적으로 요청하기 전까지 수정하지 않는다.
* 서버 연동보다 로컬 시연 기능을 우선한다.
* 네트워크 관련 변경이 필요한 경우, 먼저 사용자가 원하는 메시지 구조와 담당 범위를 확인한다.
* 서버 코드 변경이 필요한 경우 기존 배달게임 UDP 서버 구조를 먼저 확인하고, 임의로 새 네트워크 구조를 만들지 않는다.

## 씬 작업 규칙

* 사용자가 명시한 씬을 최우선으로 한다.
* 사용자가 씬을 명시하지 않으면 최근 시연 대상인 `99999` 씬을 우선 확인한다.
* `6974` 씬은 기존 통합 테스트 씬으로 참고하거나, 사용자가 요청한 경우 함께 맞춘다.
* 기존 씬을 임의로 삭제하거나 이름 변경하지 않는다.
* 씬 파일을 수정할 때는 어떤 씬을 수정했는지 완료 보고에 반드시 포함한다.
* 데모 전용 기능은 기존 핵심 시스템에 억지로 섞기보다 `Local`, `Prototype`, `Demo` 성격의 별도 스크립트로 분리한다.

## 스테이징 및 실제 프로젝트 반영 규칙

* 실제 작업은 스테이징/검증용 복사본에서 먼저 수정한다.
* 스테이징 복사본에서 컴파일 검증을 끝낸 뒤 실제 프로젝트에 반영한다.
* 실제 프로젝트 경로에 반영할 때는 사용자의 명시 승인을 받은 뒤에만 복사한다.
* 실제 프로젝트로 복사하기 전에 변경 파일 목록을 확인한다.
* `Library`, `Temp`, `Logs`, `obj`, `.vs`, `UserSettings` 같은 생성 파일은 복사하거나 커밋 대상으로 삼지 않는다.

---

## 컴파일 및 검증 규칙

* Unity 에디터 batchmode만 믿지 않는다.
* 프로젝트가 열려 있어 batchmode가 막히면, 에디터 실행에 집착하지 말고 코드 컴파일 검증으로 대체한다.
* Unity Editor batchmode가 실패하거나 프로젝트가 잠겨 있으면, Unity 설치 경로의 Roslyn/csc 또는 MonoBleedingEdge 기반으로 `Assembly-CSharp.csproj` 정적 컴파일을 시도한다.
* 기존 경고 중 `FindObjectsSortMode obsolete` 또는 기본값 필드 경고는 새 오류가 아니면 blocker로 보지 않는다.
* 새로 발생한 C# 컴파일 오류, Missing Type, Missing Namespace, API 오타는 반드시 수정한다.
* Unity 실행이 필요한 런타임 검증은 완료 보고에서 “Unity에서 직접 확인 필요”로 분리해서 적는다.

## 지형 시스템 주의사항

* 지형 생성/파괴는 다음 데이터가 서로 어긋나지 않게 유지한다.

```text
runtimeVisualTexture
collisionMask
solidMask
terrainTypeMask
Dirty Chunk
BoxCollider2D
```

* 지형 생성은 화면에만 보이고 Collider가 없거나, Collider만 있고 화면에 안 보이는 상태가 되면 안 된다.
* 지형 파괴 후 Collider 찌꺼기가 남아 캐릭터가 공중에 걸리지 않도록 한다.
* 청크 경계에 걸친 생성/파괴는 관련 청크를 모두 Dirty 처리한다.
* Created 지형은 다시 파괴 가능해야 한다.
* 캐릭터 Collider와 겹치는 픽셀은 생성하지 않는다.
* 캐릭터를 완전히 밀어내거나 끼우는 방식으로 지형을 생성하지 않는다.

## 코드 스타일

* 현재 프로젝트의 네이밍과 구조를 우선한다.
* 임의로 대규모 아키텍처를 교체하지 않는다.
* 새 클래스가 필요할 경우 역할이 명확한 이름을 사용한다.
* 값은 가능하면 Inspector에서 조절 가능하게 `[SerializeField]`로 둔다.
* 공개 API가 필요할 때만 `public`을 사용하고, 내부 값은 가급적 `private` 또는 `private serialized field`로 둔다.
* 기존 기능과 충돌할 수 있는 변경은 옵션으로 켜고 끌 수 있게 만든다.
